# SweetHome — Product Requirements Document (PRD)

**Версия:** 1.0  
**Дата:** 2026-05-02  
**Платформа:** Kotlin Multiplatform (Android + iOS)  
**Бэкенд:** Go microservices + PostgreSQL  

---

## 1. О продукте

### Vision
SweetHome — мобильное приложение для управления жизнью: домашние дела, задачи, покупки, привычки и цели — всё в одном месте. Подходит для индивидуального использования, семей и рабочих групп.

### Ключевая ценность
- Разделение личного и рабочего без переключения между приложениями
- Умная подача задач: приложение знает, в каком контексте ты сейчас (работа / дом)
- Семейное пространство с геймификацией и наградами для детей
- Простота: создание задачи в 2–3 тапа

### Целевая аудитория
- Взрослые 20–40 лет, ведущие активный образ жизни
- Семьи с детьми (домашние дела, награды, распределение обязанностей)
- Работающие люди, которым важно разделять рабочее и личное

## 2. Аутентификация и аккаунты

### 2.1. Типы аккаунтов

| Тип | Описание |
|-----|----------|
| **Полный** | Email + пароль; или OAuth Google / Apple |
| **Гостевой** | Создаётся при первом запуске без данных. Позже привязывается к email или OAuth |

### 2.2. Регистрация и онбординг

- Email + пароль: требует принятия Terms of Service
- OAuth (Google/Apple): ToS фиксируется автоматически
- Гостевой: ToS не требуется (применяется при апгрейде)
- **При регистрации автоматически создаётся один workspace:** `personal` — «Личное»
- `work` workspace **не создаётся автоматически** — пользователь включает в Настройки → «Добавить рабочее пространство»
- Баннер для гостя о привязке email (напоминает о риске потери данных)

### 2.3. Токены

| Токен | TTL |
|-------|-----|
| Access token (JWT HS256) | 15 минут |
| Refresh token (opaque) | 30 дней, ротация при каждом обновлении |

### 2.4. Удаление аккаунта

- Доступно в Профиль → Настройки → Danger Zone
- Destructive confirmation: ввод email для подтверждения
- Полное удаление: все данные пользователя, токены, устройства
- **Кнопка недоступна офлайн** — tooltip «Нет подключения к сети»
- Соответствие GDPR

## 3. Пространства (Workspaces)

Workspace — единый контейнер для списков. Все списки принадлежат workspace.

### 3.1. Типы workspace

| Тип | Описание |
|-----|----------|
| `personal` | Личное пространство. Создаётся автоматически при регистрации |
| `work` | Рабочее пространство. Создаётся вручную через Настройки → «Добавить рабочее пространство» |
| `group` | Совместное пространство (соседи, команда) |
| `family` | Семейное пространство с геймификацией и кастомной валютой |
| `mentoring` | Наставник + ученик(и). Наставник назначает задания и устанавливает награды |

### 3.2. Роли участников

| Роль | Описание |
|------|----------|
| `owner` | Создатель workspace. Может передать владение |
| `member` | Стандартный участник |
| `mentor` | В `mentoring`: создаёт задания, устанавливает reward |
| `mentee` | В `mentoring`: выполняет задания, получает награды |

Будущие роли (MMP): `admin`, `child`

> **Prizes в family workspace:** добавлять/редактировать призы могут только `owner` и `admin`.

### 3.3. Инвайты

- 6-символьный invite-код (alphanumeric, TTL 7 дней)
- Вступление: `POST /v1/workspaces/join-by-code`
- Deep link: `sweethome://invite/{token}`
- При удалении участника: `assignedTo` во всех его элементах сбрасывается

### 3.4. Рабочие часы (Work Workspace)

- В настройках рабочего workspace: `workHoursStart` и `workHoursEnd` (время суток)
- **Часовой пояс:** локальное время устройства (не UTC). При смене TZ контекст пересчитывается автоматически
- Используется главным экраном для умного переключения контекста
- Настраивается в Профиль → Настройки → Рабочее пространство

## 4. Главный экран (Dashboard)

Главный экран — центральная точка входа. Показывает то, что важно прямо сейчас.

### 4.1. Контент экрана

1. **Контекст-пилюля** (вверху): `Работа` / `Личное` — автоматически по рабочим часам, переключается вручную. Скрывается если `work` workspace не создан
2. **Задачи на сегодня** — элементы с `dueAt = today` или просроченные (`dueAt < today`), из пространства текущего контекста. Показываются **первые 3** по приоритету (просроченные → high → medium → low), кнопка «Ещё N →» раскрывает остальные
3. **Прогресс целей** (MMP scope — placeholder в MVP) — виджет с надписью «Скоро» (не блокируется замком)
4. **Быстрые действия** — FAB + shortcuts: «Добавить задачу», «Список покупок», «Домашнее дело»

### 4.2. Логика умного контекста

- Если `work` workspace не создан → контекст-пилюля скрыта, всегда показываются personal-задачи
- Если текущее время попадает в рабочие часы → контекст «Работа», показываются work-задачи
- Иначе → контекст «Личное», показываются personal-задачи
- Пользователь может переключить вручную — ручное состояние сохраняется **до следующего автоматического переключения** (когда наступают рабочие часы или они заканчиваются)

## 6. Элементы списков (Items)

### 6.1. Общие поля

| Поле | Тип | Описание |
|------|-----|----------|
| `title` | TEXT | Название |
| `note` | TEXT? | Заметка |
| `priority` | ENUM? | `high` / `medium` / `low` |
| `isDone` | BOOL | Выполнен |
| `doneAt` | TIMESTAMP? | Время выполнения |
| `assignedTo` | UUID? | Ответственный участник workspace |
| `dueAt` | TIMESTAMP? | Дедлайн |
| `isFavorite` | BOOL | Избранное |
| `reward` | TEXT? | Награда (в `family` и `mentoring` workspace) |
| `version` | INT | Версия для LWW-синхронизации |

**Отображение:** имя исполнителя (не UUID) рядом с задачей. Просроченные (`dueAt < now`) выделяются цветом.

### 6.2. Shopping-поля

| Поле | Описание |
|------|----------|
| `quantity` | Количество |
| `unit` | Единица измерения |
| `category` | Категория товара |
| `brand` | Бренд / марка товара (**MVP**) |
| `productUrl` | Ссылка на товар (**MVP**) |
| `imageUrl` | Фото товара (**MMP** — требует CDN на бэкенде) |

Детали (brand, productUrl) отображаются в строке через индикатор chevron `›` справа. Тап → bottom sheet с деталями и кнопками «Редактировать» / «Удалить».

### 6.3. Chore-расписание

| Поле | Описание |
|------|----------|
| `intervalDays` | Повторять каждые N дней |
| `daysOfWeek` | Дни недели: `["mon","wed","fri"]` |
| `startDate` | Начало |
| `endDate` | Конец (опционально) |
| `lastDoneAt` | Последнее выполнение |
| `category` | Категория дела |

**UI выбора режима (radio, один за раз):**
- «Одноразово» — `intervalDays` и `daysOfWeek` не заполняются
- «Каждые N дней» — заполняется `intervalDays`, `daysOfWeek` игнорируется
- «По дням недели» — заполняется `daysOfWeek`, `intervalDays` игнорируется

Отображение: «осталось N дней» + цветовой индикатор (зелёный / жёлтый / красный).

### 6.4. Media-поля

| Поле | Значения |
|------|---------|
| `mediaType` | `book` / `movie` / `series` / `game` / `article` / `podcast` / `other` |
| `status` | `want` / `in_progress` / `done` |
| `url` | Ссылка |
| `rating` | 1–5 |
| `author` | Автор / режиссёр |

### 6.5. Wishlist-поля

| Поле | Описание |
|------|----------|
| `url` | Ссылка на товар |
| `imageUrl` | Изображение |
| `price` | Цена |
| `isClaimed` | Занят ли подарок (API не раскрывает кем) |
| `claimedByName` | Имя занявшего (для анонимных) |
| `isAnonymous` | Анонимное занятие |

## 7. Публичный Wishlist

1. Создать список типа `wishlist`, включить `isPublic: true` → сервер генерирует `publicToken`
2. Публичная ссылка: `/public/wishlists/{token}` — доступна без авторизации
3. Любой может «занять» подарок — `isClaimed` становится `true`
4. Повторное занятие невозможно (`409 Conflict`)
5. API не раскрывает кто занял — сюрприз сохраняется
6. Отключение: `isPublic: false` — токен сохраняется, ссылка перестаёт работать

## 9. Геймификация и Кастомная Валюта (Family Workspace)

Доступно только в workspace типа `family`.

### 9.1. HomeCurrency

- Название валюты (например, «Монеты», «Звёздочки», «Кексы»)
- Иконка (в будущем кастомная, сейчас из набора)
- Настраивается владельцем family workspace

### 9.2. Механика наград

- При создании задачи в family workspace указывается `reward` — количество валюты
- При выполнении задачи участник получает указанное количество монет
- Накопленные монеты отображаются в профиле участника

### 9.3. Reward Shop

- Список призов (название, цена в монетах)
- **Добавлять/редактировать призы могут только `owner` и `admin`** (admin — роль MMP)
- Участник тратит монеты на приз
- MMP: история транзакций

### 9.4. Семейный рейтинг

- Таблица лидеров: участники family workspace по количеству заработанных монет за период
- Периоды: неделя / месяц / всё время

### 9.5. Очки и достижения (MMP)

- Очки за выполнение задач независимо от валюты
- Достижения (бейджи)
- Рейтинг по очкам

## 10. Экран создания задачи — Динамические поля

Форма создания/редактирования задачи адаптируется под тип списка.

### 10.1. Логика видимости полей

| Условие | Скрытые поля | Показанные поля |
|---------|-------------|-----------------|
| Тип задачи: одноразовая | Расписание (intervalDays, daysOfWeek) | — |
| Тип списка: `home_chores` | — | Расписание |
| Workspace: `family` или `mentoring` | — | Поле «Награда» (reward) |
| Тип списка: `shopping` | — | Детали товара (brand, imageUrl, productUrl) |
| Тип списка: `media` | dueAt, assignedTo | mediaType, status, url, rating |

### 10.2. Шаблоны при создании задачи

- Кнопка «Использовать шаблон» в форме создания
- Открывает bottom sheet с:
  - Шаблонами chore-дел (из API `/suggestions/chore-templates`)
  - Часто добавляемыми элементами (из API `/suggestions/frequent-items`)
  - Избранными элементами
- После выбора шаблона — форма pre-fill, все поля доступны для редактирования перед сохранением

## 11. Организация Home Space — Комнаты

Для personal/family workspace типа `home_chores` и `general_todos` применяется разбивка по комнатам.

### 11.1. Комнаты по умолчанию

- Кухня, Спальня, Гостиная, Ванная, Балкон, Детская
- Возможность создать свою комнату

### 11.2. Навигация

- TabRow по комнатам в HomeScreen
- Первая вкладка всегда **«Общее»** — содержит списки без назначенной комнаты (`roomType = null`)
- Остальные вкладки — по созданным комнатам
- Каждая вкладка = списки этой комнаты

### 11.3. Фильтры (Jira-style)

Chip-фильтры под TabRow:
- По приоритету: Высокий / Средний / Низкий
- По исполнителю: Я / [имена участников]
- По статусу: Активные / Выполненные / Просроченные

### 11.4. Иконки списков

Иконка списка отображается в UI (emoji или из предустановленного набора).

## 12. Группы и Семья

В групповых и семейных списках:

### 12.2. Лента активности

- События workspace: кто создал, выполнил, добавил участника
- Отображается в GroupDetailScreen

### 12.3. Чат

- REST сообщения + статусы прочтения (MVP)
- WebSocket (MMP)

## 13. Список продуктов из Меню

Флоу создания shopping list на основе рецепта / меню:

1. В экране создания списка: кнопка «Создать из меню / рецепта»
2. Пользователь вводит текст рецепта или выбирает шаблон рецепта
3. Приложение парсит ингредиенты → pre-fill элементов списка
4. Пользователь редактирует список перед сохранением

## 14. Синхронизация (Offline-first)

- Модель: **Last Write Wins (LWW)** по `version`
- Endpoint: `GET /v1/sync?since=<RFC3339>`
- Soft delete: `deletedAt` вместо физического удаления
- **MVP:** delta-sync при открытии приложения
- **MMP:** очередь отложенных операций, фоновый sync worker, sync при восстановлении сети

---

## 15. Уведомления

- **Push:** FCM (Android) + APNs (iOS), регистрация устройства при старте
- **In-app:** лента уведомлений в приложении
- **Настройки:** по каналам (выполнение задач, инвайты, напоминания о дедлайнах)
- **MMP:** push-напоминания для chores накануне или в день выполнения

## 17. Интеграция с Календарём

### MVP — экспорт в системный календарь

- Задачи с `dueAt` → единичное событие в системном календаре (Android Intent / iOS Calendar API)
- Домашние дела с расписанием → повторяющиеся события (iCalendar RRULE)

### MMP — шаринг календаря

Пользователь настраивает уровень видимости своего календаря:

| Уровень | Что видно |
|---------|-----------|
| Только занятые слоты | Блоки «занят» без деталей |
| По типу задач | Видны только задачи выбранного типа (например, только work) |
| Гибко | Пользователь выбирает конкретные списки для шаринга |

## 18. Telegram-бот (MLP)

- Привязка Telegram-аккаунта через одноразовый код в приложении
- Команды: `/start`, `/add <text>`, `/list`, `/done <id>`
- Задачи добавляются в inbox-список пользователя

## 19. Личные Цели (MMP)

Концепция для MMP scope (MVP: placeholder на главном экране):

- Создание цели: название, описание, дедлайн, список шагов
- Прогресс = % выполненных шагов
- Виджет прогресса на главном экране Dashboard
- Цели принадлежат personal workspace
